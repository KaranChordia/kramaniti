# Publishing Checklist

**Purpose:** Define the mandatory review sequence before Kramaniti content, website copy, campaign assets, or public posts are published.

**Key Findings:** [Fact] Kramaniti agents do not publish or approve public claims without human approval. [Recommendation] Publishing should be treated as a controlled workflow, not an automatic output.

**Evidence:** `AGENTS.md`, `06_ai_agent_context/system_prompts/master_context.md`, `09_reviews/proof_register.md`, and `03_brand_strategy/brand_system/brand_operating_kit.md`.

**Interpretation:** [Inference] The simplest safe workflow is draft, strategy review, proof review, asset check, founder approval, publish, then analytics log.

**Implications:** This checklist protects brand clarity, proof discipline, and founder control.

**Open Questions:** Which channels will use this checklist first?

**Next Steps:** Use this checklist for every public-facing content item before changing `Publish Status` in the content calendar.

---

## Pre-Publish Gate

- [ ] Goal is clear.
- [ ] Audience is clear.
- [ ] Content maps to a pillar or campaign.
- [ ] Language avoids generic AI-agency phrasing.
- [ ] Claims are labeled, softened, or source-backed.
- [ ] Proof and Governance Auditor reviewed proof-sensitive material.
- [ ] Asset Librarian confirmed asset source and usage status.
- [ ] Founder approved publication.
- [ ] Publication channel and date are recorded in `04_content_system/content_calendar.md`.

## Channel Checklist

| Channel | Required Checks |
| :--- | :--- |
| LinkedIn | Founder voice, clear first line, no unsupported client or metric claims, direct CTA only if relevant |
| Newsletter | Clear thesis, useful body, no inflated authority claims, source/proof notes checked |
| Website | Website Steward review, responsive QA when code changes, proof register checked |
| Insights | Source freshness, proof register checked, no repeated thesis pattern, build validation when code changes |
| Short video | Script approved, visual assets registered, no invented scenes or proof shots |

## Post-Publish Follow-Up

- [ ] Add entry to `04_content_system/analytics_log.md`.
- [ ] Record what should be repeated, revised, or stopped.
- [ ] Move any reusable decision into `03_brand_strategy/brand_system/brand_decisions.md`.
- [ ] Add proof issues or permission questions to `09_reviews/proof_register.md`.
